# Social Sharing Strategy

## Purpose

Share the process of building Knowledge Pools as an open technical journey:

- explain why basic RAG is not enough;
- show how an agent-oriented knowledge repository can be designed;
- document decisions as they happen;
- attract collaborators, critique, and future users.

## Target Readers

- AI builders working with RAG and agents.
- Developers building personal or team knowledge systems.
- Researchers interested in memory, retrieval, and verification.
- Product-minded builders looking for practical agent architectures.

## Tone

- Clear, technical, and exploratory.
- Honest about uncertainty.
- Prefer concrete progress over hype.
- Show decisions and tradeoffs, not only conclusions.

## Recommended Channels

- X / Twitter: short build-in-public updates and diagrams.
- LinkedIn: more polished progress summaries and lessons.
- GitHub README / Discussions: durable project context.
- Blog / Newsletter later: deeper architecture essays.

## Content Pillars

### 1. RAG Limitations

Explain problems such as chunk myopia, weak conflict handling, and poor temporal reasoning.

### 2. Agent Architecture

Show how ingestion, retrieval planning, reasoning, verification, and update agents cooperate.

### 3. Knowledge Modeling

Explain why claims, decisions, concepts, procedures, and questions are better durable units than raw chunks alone.

### 4. Build Logs

Share what changed in the repository and why.

### 5. Open Questions

Ask the community about unresolved design choices.

## Guardrails

- Do not claim production readiness until there is a working MVP.
- Do not imply that agents eliminate the need for evidence.
- Do not frame vector search as useless; frame it as insufficient alone.
- Do not publish private notes, credentials, or user-specific data.
